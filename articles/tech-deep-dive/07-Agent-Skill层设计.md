# Agent Skill 层设计：让能力像乐高一样可复用

> 🎯 **知识地图位置**：第5层 → 业务 Agent 深度开发 ｜ 📊 **难度**：进阶
>
> 这是「Agent技术笔记」的第七篇文章。前面六篇从决策推理、记忆管理、工具调用到 LangGraph 框架，解决的都是 Agent 「底层怎么跑」的问题。但从这一篇开始，我们进入**真正决定 Agent 好不好用的环节**——**Skill（技能）层设计**。

---

## 先说结论

Skill 解决的核心问题是：

> **如何让 Agent 的能力「模块化、可复用、渐进式加载」，而不是把全量知识一股脑塞给大模型。**

很多人搭建 Agent 时容易掉进一个误区：把所有业务知识、规则、工具都写进 system prompt，期望模型"记住"一切。结果呢？上下文臃肿、响应变慢、模型注意力被稀释，最终效果反而变差。

Skill 就是解药——它把特定场景下的**执行逻辑、参考资料、工具调用**封装成独立模块，**想用的时候才加载**，用完即走，彻底告别 context 膨胀。

---

## 一、用炒菜理解 Skill：一个接地气的类比

Skill 到底是什么？我们用炒菜来做个类比，一下就懂了。

| 炒番茄蛋 | Agent Skill |
|:---|:---|
| **菜谱（流程）**：起锅烧油→先炒鸡蛋→再炒番茄→最后混合调味，出锅 | **指令层（Kernel）**：定义执行步骤和决策逻辑的 Markdown 文件 |
| **配方**：番茄 300g、鸡蛋 3 个、盐 3g、糖 8g | **元数据**：Skill 的概述信息，一句话说明这个 Skill 是干什么的 |
| **工具**：炒锅、锅铲、菜刀、切菜板 | **脚本/函数**：具体执行工具（数据库查询、API 调用、计算逻辑） |
| **材料**：番茄酱 | **参考数据**：动态加载的外部知识（文档、FAQ、历史案例） |

看完这个对比，Skill 的本质就很清楚了：

> **Skill 不是工具，也不是知识库——它是一套完整的「执行说明书」，规定了在特定场景下，Agent 应该按什么步骤思考、什么时候调用什么工具、参考哪些资料。**

## 二、Skill 的三层结构：元数据、指令层、资源层

理解了 Skill 的本质，再来看看它的具体构造。一个标准 Skill 包含三层：

```
Skill/
├── SKILL.md          # ① 元数据 + 指令层（Kernel）
├── references/       # ② 参考数据（动态加载）
│   ├── 业务流程规范.md
│   └── 常见问题解答.md
├── scripts/          # ③ 工具脚本（函数/代码）
│   ├── search_cargo.py
│   └── price_calc.py
└── assets/           # ④ 辅助资源（图片、模板等）
    └── 报价单模板.json
```

### 2.1 元数据：Skill 的"身份证"

元数据通常只占用几十字，但它**至关重要**——因为它是模型做 Skill 路由决策时的唯一依据。

```markdown
<!-- SKILL.md 头部示例 -->
## 元数据
- 名称：货源搜索与推荐
- 描述：当司机需要找货源时，根据司机的当前位置、偏好路线和货物类型，搜索并推荐合适的货源
- 触发条件：用户输入包含"找货"、"货源"、"拉货"等关键词
- 优先级：高
```

> 💡 **为什么元数据如此关键？**
>
> Agent 面对用户输入时，第一步就是判断「这个需求该交给哪个 Skill 处理」。模型看不到 Skill 的完整内容，只能看到元数据里的**名称和描述**。如果描述写得不准，模型就会「选错 Skill」，后面执行得再好也白搭。

### 2.2 指令层（Kernel）：执行逻辑的"大脑"

指令层是一个 Markdown 文件，规定了 Skill 内部**完整的执行逻辑**。它告诉 Agent：

1. **场景理解**：当前任务的目标和约束
2. **执行步骤**：分几步完成，每步做什么
3. **决策点**：什么时候调用工具、什么时候参考外部资料
4. **输出格式**：最终结果应该长什么样

以一个 Skill 指令层为例：

**执行步骤：**
1. **意图确认**：确认用户需求，明确关键参数
2. **参数收集**：收集用户的偏好和约束条件
3. **资源搜索**：调用搜索脚本，传入参数进行搜索
4. **结果筛选**：根据匹配度排序，过滤掉不符合条件的结果
5. **结果呈现**：以推荐列表形式输出

**决策规则：**
- 如果搜索结果为空 → 扩大搜索范围，重新搜索
- 如果结果不满足用户期望 → 提示"当前条件匹配度偏低"，仍展示结果
- 如果匹配结果超过 5 条 → 展示 Top 5，其余放入"更多推荐"

**输出格式：**
```json
{
  "recommendations": [...],
  "note": "...",
  "action_needed": "..."
}
```

### 2.3 资源层：按需加载的"弹药库"

资源层包含 Skill 执行过程中可能需要调用的**一切外部资源**，分为三类：

| 资源类型 | 内容 | 加载时机 |
|:---|:---|:---|
| **参考资料**（references/） | 业务规范、FAQ、操作手册 | 执行到对应步骤时按需加载 |
| **工具脚本**（scripts/） | 函数脚本、API 封装 | 指令层明确调用时加载 |
| **辅助资源**（assets/） | 模板文件、配置文件 | 格式化输出时按需引用 |

## 三、渐进式加载：Skill 最核心的设计哲学

Skill 之所以强大，**不在于它能装多少东西，而在于它知道「什么时候用什么东西」**。

> 🧠 **渐进式加载的本质**：不是把 Skill 里所有内容一次性塞进上下文，而是让 Kernel 作为"指挥官"，**执行到哪一步，才加载那一步需要的资源**。

### 对比：全量加载 vs 渐进式加载

| 方式 | 上下文大小 | 加载时间 | 效果 |
|:---|:---|:---|:---|
| **全量加载** | 把所有 Skill 内容一次性塞给模型 | 上下文臃肿（5k+ tokens）| 慢，模型注意力分散 |
| **渐进式加载** | 只加载当前步骤需要的资源 | 上下文精简（1-2k tokens）| 快，模型聚焦当前任务 |

举个例子：一个包含 20 个货源搜索规则的 Skill，全量加载时 model 要记住全部规则才能开始搜索。而渐进式加载只告诉 model「第一步：确认意图」，确认完意图后才加载「第二步：参数收集」的规则，以此类推。

```python
# 渐进式加载的伪代码示意
def execute_skill(skill, user_input):
    # 第1步：加载 Kernel 的入口逻辑
    kernel = load_skill_kernel(skill.SKILL_md)
    
    while not task_complete:
        # 只加载当前步骤需要的资源
        step_instruction = kernel.get_current_step()
        references = load_references(step_instruction.required_refs)
        tools = load_tools(step_instruction.required_tools)
        
        # 构造精简上下文，交给模型执行
        context = build_context(step_instruction, references, user_input)
        response = model.invoke(context)
        
        # 根据响应推进到下一步
        kernel.advance(response)
    
    return response
```

> 💡 **渐进式加载的最大价值**
>
> 1. **控制上下文长度**：避免 token 浪费和模型注意力稀释
> 2. **提升响应速度**：减少每次调用的上下文量
> 3. **降低认知负担**：模型只需关注当前步骤，不用背整本说明书
> 4. **方便调试定位**：哪一步出了问题，一目了然

## 四、开源 Skill 生态：站在巨人的肩膀上

Skill 的另一个巨大价值在于**可复用性**。

市面上已经有很多高质量的开源 Skill，它们本质上就是**特定场景下沉淀下来的执行逻辑包**。你不需要自己从零写一套"怎么写代码"的 Skill，直接用开源的即可；同理，法律、医疗、金融等领域都有成熟的开源 Skill 可以参考。

| 开源 Skill 示例 | 场景 | 价值 |
|:---|:---|:---|
| **代码生成 Skill** | 根据需求生成代码 | 封装了代码规范、最佳实践、常见错误处理 |
| **数据分析 Skill** | 读取数据、生成报表 | 封装了数据清洗、可视化、洞察提取的流程 |
| **客服对话 Skill** | 处理用户咨询 | 封装了意图识别、FAQ 匹配、转人工的决策链 |

> 💡 **为什么开源 Skill 能"插拔式"使用？**
>
> 因为 Skill 是**场景隔离**的——每个 Skill 只负责一个明确的能力域，彼此之间通过标准接口交互。就像乐高积木，每个积木（Skill）有自己的形状（接口规范），可以独立使用，也可以组合搭建更复杂的结构。

## 五、实践体会：为什么 Skill 范式能加速迭代

在实际的 Agent 项目实践中，我们采用 Skill 范式设计框架后，一个明显的感受是：**迭代效率显著提升**。

为什么？因为 Skill 天然适配了业务 Agent 的核心挑战——**场景划分**。

### Skill 范式的迭代飞轮

```
发现问题 ──→ 定位 Skill ──→ 调整内容 ──→ 验证效果
    ↑                                      │
    └──────────── 快速循环 ◄───────────────┘
```

具体来说：

| 问题类型 | Skill 范式的应对方式 | 迭代速度 |
|:---|:---|:---|
| **场景划分不合理** | 增删或合并 Skill，调整路由规则 | ⭐⭐⭐⭐⭐ 快 |
| **某个场景效果不好** | 只修改对应 Skill 的指令层 | ⭐⭐⭐⭐⭐ 快 |
| **新增业务场景** | 新增一个 Skill，不影响现有能力 | ⭐⭐⭐⭐⭐ 快 |
| **模型升级/切换** | Skill 层无感知，底层框架适配即可 | ⭐⭐⭐⭐ 较快 |

> 💡 **关键洞察**
>
> 在实际业务中，决定 Agent 效果好坏的往往不是底层框架（ReAct、LangGraph 这些只要搭好就稳定了），而是**场景划分是否清晰、每个场景下的 Skill 内容是否到位**。
>
> Skill 范式把「场景」和「内容」解耦——场景变了调路由，内容变了改 Skill，两者互不影响，迭代路径非常清晰。

### 一个真实小场景的迭代案例

我们负责的一个「调度 Agent」，最初只有一个大 Skill 包揽所有逻辑。效果不理想，排查后发现：

1. **用户主动发起** 和 **系统主动推荐** 是两个完全不同的场景，混在一起导致决策逻辑打架
2. **新用户** 和 **老用户** 的偏好模型差异很大，但 Skill 里没有区分

用 Skill 范式重构后：

```
调度 Agent
├── Skill-主动搜索          ← 用户发起，强调精准匹配
├── Skill-系统推荐          ← 系统推送，强调发现潜在机会
├── Skill-新用户引导        ← 新手专属，简化流程、增加引导
└── Skill-老用户智能推荐     ← 老客专属，基于历史偏好深度推荐
```

调整后，场景划分清晰了，每个 Skill 只负责一个明确目标，内容聚焦了，效果自然提升。而且后续想增加「议价 Skill」或者「路线规划 Skill」，直接插拔即可。

## 六、Skill 设计的最佳实践

基于实践总结几条 Skill 设计的原则：

### 1. 一个 Skill 只负责一个明确的场景

> ❌ 反例：一个 Skill 既管找货又管议价又管下单——场景混杂，逻辑纠缠，效果差。
> ✅ 正例：「找货 Skill」只管搜索和推荐，「议价 Skill」只管价格协商，各司其职。

### 2. 元数据要写得像「给模型看的简历」

模型靠元数据判断用哪个 Skill。描述要**准确、具体、有区分度**，避免模糊词汇。

> ❌ "处理用户请求" —— 太泛
> ✅ "当用户需要搜索货源并获取推荐列表时触发" —— 具体明确

### 3. 指令层遵循「单一职责 + 渐进式展开」

不要在一个步骤里塞太多逻辑。每个步骤只做一件事，做完再加载下一步的资源。

### 4. 工具脚本要「无状态、纯函数」

Skill 里的工具脚本应该是**幂等**的——同样的输入永远返回同样的输出，不依赖外部状态。这样 Skill 才能被安全地复用和组合。

### 5. 参考资料要「版本化管理」

业务规则经常变，参考资料也要跟着更新。建议用 Git 管理 Skill，每次规则变更都有迹可循。

## 总结

| 维度 | 没有 Skill 层 | 有 Skill 层 |
|:---|:---|:---|
| **上下文管理** | 全量塞入，臃肿膨胀 | 渐进加载，按需取用 |
| **场景隔离** | 所有逻辑混在一起 | 每个 Skill 独立负责一个场景 |
| **迭代效率** | 一改全改，牵一发动全身 | 定位精准，只改对应 Skill |
| **复用性** | 每个项目从零写 | 开源 Skill 直接插拔使用 |
| **可解释性** | 黑盒，不知道为啥这么答 | 白盒，每个步骤逻辑清晰 |

Skill 层是连接「底层框架能力」和「上层业务价值」的关键桥梁。前面几篇文章讲的 ReAct、Memory、Tool Calling、LangGraph——这些都是让 Agent 「能跑起来」的基础设施。而 **Skill 层决定的是 Agent 「跑得好不好」**。

场景划分清晰了，Skill 内容打磨到位了，Agent 的效果就不会差。

> 🎯 **一句话记住 Skill**
>
> **Skill 不是知识库，不是工具集——它是特定场景下的「执行说明书」，渐进式加载，按需组合，像乐高一样可复用。**
