# Memory 模块设计实录：如何让 Agent 记住三个月前的对话

> 🧠 **知识地图位置**：第3层 → 记忆管理模块 ｜ 📊 **难度**：进阶
>
> 本文是「Agent技术笔记」关于 **Agent 记忆管理**的系统性解析。记忆，是 Agent 区别于普通 LLM 调用的核心能力之一——没有记忆的 Agent，每一次都是从零开始；有了记忆，Agent 才能真正意义上"成长"。本文从概念到工程实现，逐层拆解记忆模块的设计逻辑与关键决策。

## 先说结论

记忆模块的本质，不是"让 Agent 记东西"这么简单。

它是一套完整的**信息生命周期管理系统**：哪些信息值得被记录？以什么形式存？什么时候用？怎么保证新鲜度和可用性？每一个环节都会直接影响 Agent 的效果和成本。

## 一、什么是记忆？从人说起

在讲 Agent 的记忆之前，先从人的角度理解这个概念。

**对人而言，记忆是"在大脑中留存下来的，曾经发生过的事"。**

注意这里有一个隐含条件：**不是所有发生过的事，人都能记住**。你今天刷了几十条短视频，大概率明天就全忘了；但你第一次找到工作时老板打来电话的那个下午，可能几年后还清晰可见。

大脑对记忆有一套天然的"筛选机制"——情绪浓度高的、重复出现的、与当前目标强相关的，才更容易被留存下来。剩下的，大脑会选择性遗忘。这不是缺陷，而是为了让有限的认知资源聚焦在真正重要的事情上。

**对 Agent 而言，记忆的定义要宽泛得多：**

> Agent 的记忆，是**它与外界交互过程中产生的所有数据**——输入（用户指令、环境观察）、过程数据（推理链、中间结果、工具调用记录）、输出（最终答案、执行结果）。

两者的对比很有意思：

| 维度 | 人的记忆 | Agent 的记忆 |
|:---|:---|:---|
| **内容来源** | 感官体验、情感事件 | 所有交互产生的数据 |
| **筛选机制** | 大脑天然筛选（情绪、重复、关联度） | 需要**人为设计**筛选逻辑 |
| **存储介质** | 神经网络连接 | 向量数据库、文件系统、模型权重 |
| **容量上限** | 有限，且随时间自然衰减 | **理论上无限**（只要存储够） |
| **遗忘机制** | 自动的、被动的 | 需要**主动设计**消亡/淘汰策略 |

最后一行的差异，是整个记忆模块设计中最值得深想的地方——**人靠遗忘来减负，Agent 没有天然的遗忘，所以需要人来替它设计"该忘什么"**。

## 二、为什么需要研究记忆？

### 2.1 一个直觉类比

没有记忆的 Agent，就好比一个**什么也没经历过的孩子**。

让他去完成一件事，做完之后立刻忘了这件事；下次让他做同样的事，还是原地踏步——一样的判断，一样的错误，一样的低效。他不会从这次经历中学到任何东西，因为他根本不记得有过这次经历。

有了记忆的 Agent 则完全不同。它知道"上次做这件事，第三步的工具调用返回了空值，原因是参数格式不对"。这一次，它会提前规避；甚至主动调整策略，换一个更稳健的路径。

这就是记忆的核心价值——**让 Agent 有历史，有历史才能有成长。**

### 2.2 价值导向：记忆能带来什么

总结下来，有几个比较关键的价值点：

**① 避免重复无意义动作和信息漂移**

身份、约束、偏好、业务规则——这些信息一旦建立，就应该成为可复用的上下文，而不是每次都从零重建。没有记忆，Agent 可能在同一个任务里反复澄清同一个问题，或者在不同会话里给出矛盾的答案。

**② 任务一致性和可信度**

跨多轮次的长任务中，口径、目标、上下文必须保持一致。没有记忆的 Agent 容易在第 5 轮"忘了"第 2 轮用户说的前提条件，导致后续回答走样，用户体验急剧下降。

**③ 效率与成本**

有了记忆，可以减少不必要的澄清轮次，减少重复的工具调用。在规模化的生产环境里，这些"节省"叠加起来是非常显著的成本差异。

**④ 自我改进（经验沉淀）**

这是记忆最深层的价值。从成功和失败的轨迹里，提炼出可复用的"打法"——某类任务的最优工具调用顺序、某种报错的应对策略、某个用户的偏好模式。这些沉淀下来，就是 Agent 的"成长"。

### 2.3 工程现实：不能无限记

理论上，Agent 可以把所有发生过的交互都记下来。存储空间够大，什么都留着就行了——这听起来很美好，但完全不现实。

**工业成本 cover 不住。**

一个企业级 Agent 系统，每天可能产生几十万次交互，每次交互涉及几千 token 的上下文。全量存储是一方面，更大的问题是：每次召回时，你要从海量的历史记忆里找出真正有用的那一小片——检索成本、噪声干扰、上下文窗口占用……全都是实际问题。

> 💡 **工程核心矛盾**：记忆越多，召回的信息越丰富；但记忆越多，噪声越大、成本越高、上下文越臃肿。**如何在信息量和可用性之间找到平衡点**，是记忆模块设计的核心挑战。

这也是为什么需要认真研究记忆的**压缩、存储、更新和消费**——这些环节不是可选项，而是工业级记忆系统的必选基础建设。

## 三、Agent 的记忆类型：工程视角的分类

记忆的分类方式很多。学术上常见的分法是按时效（短期/长期），或者按认知心理学（情景记忆、语义记忆、程序性记忆）。这些分法在理论上没问题，但对于工程实现来说，指导性不够直接。

这里我更倾向于从**工程实现角度**来分，按照"记忆内容的性质和使用方式"分成三类：

### 3.1 事实记忆（Fact Memory）

**定义：可声明的、结构化的事实。**

这类记忆的特点是：内容相对稳定，可以被明确表达，是"关于世界是什么样的"的描述。

典型内容：
- 用户的身份信息、偏好设置、历史习惯
- 业务数据（某个货主的常用路线、某个司机的车型偏好）
- 环境状态（当前任务的约束条件、系统配置参数）

**工程特点**：读多写少，更新频率低，对准确性要求高（一条错误的事实记忆，可能污染后续所有决策）。适合用结构化存储（关系型数据库、图数据库）或 K-V 存储。

### 3.2 经验记忆（Episodic / Skill Memory）

**定义：行为轨迹沉淀出来的改进信号和可复用打法。**

这类记忆的特点是：来源于过去的执行过程，是"关于如何做事的"的总结。

典型内容：
- 过去某类任务成功/失败执行的完整轨迹快照
- 总结出来的策略和技巧（"处理这种报错，先做 X 再做 Y"）
- 工具调用的最优参数组合

**工程特点**：写入频率中等，读取时需要语义匹配（当前任务与过去经验的相关性判断），更新机制复杂（成功经验要强化，失败经验要保留用于避错，过时经验要淘汰）。适合向量数据库存储 + 语义检索。

> 💡 **Reflexion 模式的本质**，就是让 Agent 在每次执行失败后，把失败原因和应对策略写入经验记忆，下次遇到类似情况时主动召回避开——这也是为什么经验记忆是「自我改进」价值的核心载体。

### 3.3 工作记忆（Working Memory）

**定义：当前工作周期内的临时状态数据。**

这类记忆的特点是：强时效性，任务结束后通常不再需要（部分会被蒸馏为经验记忆留存下来）。

典型内容：
- 当前轮次的推理链（Thought-Action-Observation 序列）
- 多轮对话中的关键状态变化（"用户在第 3 轮改变了目标"）
- 子任务的中间计算结果

**工程特点**：读写频率最高，时效最强，容量需求最大（因为要承载整个推理过程的中间状态）。适合放在内存缓存中，配合 context window 管理策略。

---

三类记忆的核心差异一览：

| 记忆类型 | 内容性质 | 时效性 | 存储介质 | 典型使用场景 |
|:---|:---|:---|:---|:---|
| **事实记忆** | 结构化事实 | 长期稳定 | 关系型/K-V 数据库 | 用户身份、业务规则 |
| **经验记忆** | 行为轨迹沉淀 | 中长期 | 向量数据库 | 打法复用、避错策略 |
| **工作记忆** | 当前任务状态 | 短期 | 内存缓存 | 推理链、中间结果 |

## 四、记忆的存储形式

记忆的存储形式，这里参照论文 **《Memory in the Age of AI Agents: A Survey》** 里的分类框架，从广义视角大致分为三种形式。

### 4.1 Token-level（显式文本）

最直观的存储形式。记忆以**可读的文本片段**形式保存，比如对话历史、日志、RAG 检索结果、总结摘要等。

**优点**：
- 可解释性最强——你能直接读出记忆里存了什么
- 易于调试和审计（出了问题，打开记忆文件就能看到）
- 更新方便，增删改都在文本层面操作

**缺点**：
- 占用 context window（每次召回都要把文本塞进 Prompt，消耗 token）
- 长度有上限，海量历史需要压缩和筛选

> 实际工业场景里，**Token-level 是使用最广泛的形式**，原因就是可解释性强。大部分企业对 AI 决策链路都有可审计的要求，而 token-level 的记忆天然满足这个需求。

### 4.2 Parametric（参数化）

记忆被**写进模型权重**里，而不是存在外部存储中。微调（SFT）、LoRA、知识蒸馏，都属于这一类。

**优点**：
- 不占用 context window（知识内化在模型参数里，不需要显式召回）
- 对垂域任务效果非常好（模型"天生"就懂这个领域）

**缺点**：
- 可解释性偏低（模型"知道"某件事，但你无法直接看到这条知识存在哪里）
- 更新成本高（更新一条记忆 = 重新微调模型，代价大）
- 容易遗忘（新微调可能覆盖旧知识，即"灾难性遗忘"问题）

> 在我们的业务实践中，Parametric 形式主要用于"稳定的领域知识注入"——比如物流行业的专业术语、货运定价逻辑，这些不会频繁变化的知识，通过 SFT 写进模型效果很好；但对于需要动态更新的业务数据，就不适合用这种形式。

### 4.3 Latent（隐式表征）

最隐式的存储形式。记忆以**隐藏的向量表征**存在，比如 KV Cache 缓存、隐空间的记忆压缩表示等。

**优点**：
- 计算效率高（不需要文本解析，直接在表征层面操作）
- 适合超长上下文的压缩存储

**缺点**：
- 可解释性最弱（你根本看不出表征里存了什么）
- 工程实现复杂，调试困难
- 目前工业落地成熟度相对较低

---

三种存储形式的对比：

| 存储形式 | 可解释性 | 更新成本 | Context 占用 | 工业落地成熟度 |
|:---|:---:|:---:|:---:|:---:|
| **Token-level** | ⭐⭐⭐⭐⭐ 最强 | 低 | 高（占 Prompt） | ⭐⭐⭐⭐⭐ 最主流 |
| **Parametric** | ⭐⭐ 偏低 | 高（需微调） | 无 | ⭐⭐⭐⭐ 成熟 |
| **Latent** | ⭐ 最弱 | 中 | 低 | ⭐⭐ 仍在探索 |

**实际工业选择的优先顺序**：Token-level 优先（可解释性最强，可审计），其次是 Parametric（垂域效果好，不消耗 context），Latent 用在特定的效率优化场景。

## 五、记忆模块设计的关键环节

知道了"记忆是什么"和"用什么存"，下一步才是真正的工程设计。

记忆模块的生命周期，核心可以抽象为四个环节：**写入 → 治理 → 检索 → 使用**。每一个环节都藏着真正的设计难点。

```
[交互产生数据]
      │
      ▼
┌──────────────┐
│   写入 Write  │  ← 该记什么？怎么压缩？怎么结构化？
└──────┬───────┘
       │
       ▼
┌──────────────┐
│  治理 Govern  │  ← 怎么更新？怎么淘汰过期记忆？怎么处理冲突？
└──────┬───────┘
       │
       ▼
┌──────────────┐
│  检索 Recall  │  ← 什么时机触发？怎么精准召回？怎么控制噪声？
└──────┬───────┘
       │
       ▼
┌──────────────┐
│   使用 Use    │  ← 怎么组装进 Prompt？怎么平衡信息量与 token 预算？
└──────────────┘
```

### 5.1 写入（Write）

**核心问题：记什么？不记什么？怎么记？**

写入看似简单，其实是整个链路里最容易被忽视、却最影响后续效果的环节。

**① 筛选：并非所有交互都值得写入**

一次完整交互可能产生大量数据——用户的话、LLM 的推理链、工具调用记录、中间结果……如果全量写入，噪声极大。

实践中比较有效的做法，是建立一套**记忆写入策略**：

- **关键事实**（用户偏好变化、新业务规则）→ 写入事实记忆
- **完整执行轨迹**（特别是失败案例和高价值成功案例）→ 写入经验记忆
- **当前任务状态**（推理链、中间结果）→ 写入工作记忆（任务完成后选择性蒸馏）

> 💡 **双模型写入**是一个工程上常用的方案：用小模型（8B 量级）做"值不值得记"的分类判断，再用大模型做结构化提取——这样既控制了成本，又保证了写入质量。（在上一篇讲决策机制时我们也提到了这个双 LLM 架构）

**② 结构化：记忆的格式决定了它能否被有效召回**

同样是"用户喜欢走北上广三角区的路线"，以下两种存法效果完全不同：

```
❌ 原始存法（噪声大，难召回）：
"2024年3月12日，用户张三在聊天里提到他一般都跑北上广那边的货，
 感觉那边货多好找，所以路线偏好是北上广一带"

✅ 结构化存法（干净，易检索）：
{
  "entity": "用户_张三",
  "attribute": "路线偏好",
  "value": ["北京", "上海", "广州"],
  "confidence": 0.9,
  "source": "对话_20240312",
  "expiry": null
}
```

结构化不是多此一举，而是为后续的精准检索打好基础。

**③ 压缩：长对话的摘要策略**

工作记忆里的推理链往往很长。如果需要把它蒸馏进长期记忆，需要设计合适的摘要策略：

- **滚动摘要（Rolling Summary）**：每隔 N 轮对话，对前 N 轮做摘要压缩，只保留关键信息
- **关键帧提取**：识别对话中的"转折点"（用户改变了目标、出现了新约束、发生了关键错误），只保留这些关键帧

---

### 5.2 治理（Govern）

**核心问题：怎么保持记忆的新鲜度和准确性？**

记忆写进去容易，但写进去之后不管，随着时间推移会出现三类问题：

**① 过期**：业务规则变了，但记忆里还存着旧版本

**② 冲突**：同一个事实在不同时间被记录了两个互相矛盾的版本

**③ 膨胀**：记忆库持续增长，检索成本越来越高，召回质量越来越差

针对这三个问题，需要对应的治理策略：

| 问题 | 治理策略 | 说明 |
|:---|:---|:---|
| **过期** | 时间戳 + 有效期标记 | 写入时打时间戳，业务规则类记忆设置过期时间；过期后打 `EXPIRED` 标记而非直接删除（保留历史快照） |
| **冲突** | 冲突检测 + 合并/覆盖策略 | 写入新记忆前，先检索是否存在同一 entity 的相关记录；冲突时优先保留更新的、可信度更高的版本 |
| **膨胀** | 分级存储 + 定期清理 | 高频访问的记忆放热存储，低频访问的记忆归档；对极低价值的记忆执行软删除 |

> 💡 **一个容易忽视的问题**：很多团队设计记忆系统时，只考虑了写入和检索，完全没有设计治理策略。结果六个月后，记忆库里躺着大量过期的、矛盾的、低价值的数据，每次召回都带着一堆噪声——这不叫"记忆"，叫"信息垃圾场"。

---

### 5.3 检索（Recall）

**核心问题：在正确的时机，从正确的地方，召回正确的记忆。**

检索是记忆消费的入口，也是最容易出性能瓶颈的环节。

**① 什么时机触发检索？**

不是所有任务都需要做记忆检索。无脑地每次对话都走一遍检索，既增加延迟，又引入不必要的噪声。

比较合理的触发策略：

- **任务开始时**：检索用户偏好记忆 + 相关经验记忆（设置好本轮任务的上下文背景）
- **工具调用失败时**：检索历史失败案例 + 对应的应对策略
- **发现新信息时**：检索事实记忆，判断是否与已有知识冲突

**② 怎么提高检索精准度？**

单纯的语义相似度检索（纯向量检索）有一个问题：它检索的是"语义相近的"，但有时候你需要的是"时间上最新的"或者"和当前实体强相关的"。

工业上比较成熟的方案是**混合检索**：

```python
def hybrid_recall(query: str, context: dict) -> list[MemoryItem]:
    """
    混合检索策略：语义相似度 + 关键词过滤 + 时间衰减
    """
    # Step 1: 向量语义检索（召回语义相关的候选集）
    semantic_candidates = vector_db.search(
        query_embedding=embed(query),
        top_k=20
    )
    
    # Step 2: 关键词过滤（缩小候选范围，提升相关性）
    entity_filtered = [
        m for m in semantic_candidates
        if context.get("entity_id") in m.related_entities
    ]
    
    # Step 3: 时间衰减（优先近期记忆）
    def recency_score(memory_item):
        days_ago = (now() - memory_item.created_at).days
        return memory_item.semantic_score * (0.95 ** days_ago)
    
    # Step 4: 综合排序，取 top-k
    ranked = sorted(entity_filtered, key=recency_score, reverse=True)
    return ranked[:5]
```

**③ 检索结果的质量评估**

召回了 5 条记忆，但不等于这 5 条都值得放进 Prompt。在使用前，需要做一轮**相关性过滤**：是否和当前任务直接相关？可信度如何？是否已过期？

### 5.4 使用（Use）

**核心问题：怎么把召回的记忆有效地注入到当前决策中？**

召回了记忆，最终还是要组装进 Prompt 里，才能被 LLM 使用。这里有几个关键的工程决策：

**① 记忆的位置和格式**

不同类型的记忆，在 Prompt 里的位置和格式不同：

```
[System Prompt]
你是一个物流 Agent，专门帮助司机找货。

## 用户信息（事实记忆）
- 司机：张三，10 年驾龄，大型货车
- 偏好路线：北京→上海、北京→广州
- 历史偏好：价格优先（而非时效优先）

## 历史经验（经验记忆）
- 上次为张三匹配了北京→上海的货，司机反馈"价格太低，不接"
  → 启示：张三的心理价位下限约为 ¥4500/单

[User Message]
张三：帮我找明天从北京出发的货
```

**② token 预算管理**

上下文窗口是有限资源。记忆内容 + 系统提示 + 当前对话 + 工具定义，加在一起不能超过 context window 限制。

实践中需要建立**记忆的 token 预算策略**：

- 为不同类型的记忆分配固定 token 配额（例如：事实记忆 ≤ 500 tokens，经验记忆 ≤ 800 tokens，工作记忆 ≤ 1200 tokens）
- 超出配额时，优先保留置信度最高、时效性最强的记忆片段
- 动态压缩：根据当前任务类型，动态调整各类记忆的权重

**③ 记忆注入不等于"粘贴所有记忆"**

这是一个常见误区。召回了 10 条记忆，不代表要把 10 条全部粘进 Prompt。**相关性弱的记忆不仅没有价值，还会引入噪声，让 LLM 的注意力分散**。

召回是广撒网，注入是精准打击——两个环节都不能省。

## 七、总结

这篇文章从"记忆是什么"出发，系统梳理了 Agent 记忆模块的核心设计逻辑：

| 维度 | 核心观点 |
|:---|:---|
| **概念** | Agent 的记忆本质是交互数据的全量沉淀，但"记什么"需要人为设计筛选机制 |
| **价值** | 记忆带来四类价值：避免重复、保证一致性、降低成本、支撑自我改进 |
| **分类** | 工程视角下分三类：事实记忆（稳定的事实）、经验记忆（行为轨迹沉淀）、工作记忆（当前任务状态） |
| **存储形式** | Token-level 可解释性最强是首选；Parametric 适合垂域知识注入；Latent 适合效率优化场景 |
| **设计环节** | 写入、治理、检索、使用——四个环节缺一不可，每个环节都有自己的设计难点 |

最后想说一句有点反直觉的话：

> **记忆设计的核心，不是"让 Agent 记住更多"，而是"让 Agent 在正确的时机，用正确的记忆，做出更好的决策"。** 这两件事看起来像，但工程取向完全不同。
